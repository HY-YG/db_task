# 数据库课程设计情况一览表

## 1. 项目采用的框架和技术

| 分析项 | 采用情况 | 作用说明 | 具体例子 |
| --- | --- | --- | --- |
| 后端框架 | FastAPI + Uvicorn | 提供 RESTful API、异步请求处理和接口组织能力 | `backend/main.py` 统一挂载 `auth`、`courses`、`learning`、`assignments`、`ai`、`admin` 路由 |
| 前端框架 | Streamlit | 快速构建课程、作业、通知、AI 助手等页面 | `frontend_streamlit/app.py` 负责页面路由与状态管理 |
| 数据库 | PostgreSQL（Neon） | 作为核心关系数据库，承载课程、用户、作业、学习记录和 AI 数据 | `backend/config/db_config.py` 通过 `NEON_URL` 连接 Neon PostgreSQL |
| ORM 与访问方式 | SQLAlchemy 2.x + asyncpg | 使用 ORM 建模并采用异步数据库访问，提高并发能力 | `create_async_engine(...postgresql+asyncpg://...)` |
| 数据校验 | Pydantic v2 | 对接口请求和响应进行字段校验，减少脏数据进入数据库 | `backend/schemas` 中对注册、课程、作业等请求进行约束 |
| 认证方式 | Bearer Token | 登录后为用户签发令牌，后续请求通过请求头鉴权 | `backend/utils/auth.py` 中解析 `Authorization: Bearer <token>` |
| AI 能力集成 | LangChain + langchain-openai | 支持课程问答、聊天助手、学习教练等智能功能 | `backend/services/chat_agent.py`、`coach_agent.py` |
| 向量能力 | pgvector + 向量字段/分块表 | 支持 AI 记忆和资源检索增强 | `backend/main.py` 启动时执行 `create extension if not exists vector`，`ai_memories.embedding` 使用 `Vector(...)` |
| 配置管理 | python-dotenv + `.env` | 将数据库连接等敏感配置与代码分离 | `backend/config/db_config.py` 通过 `.env` 加载 `NEON_URL` |

## 2. 数据库安全技术分析

| 安全技术 | 项目体现 | 说明 | 具体例子 |
| --- | --- | --- | --- |
| 连接安全 | 使用 PostgreSQL SSL 连接 | 当连接串中配置 `sslmode=require` 时，数据库连接启用 SSL | `backend/config/db_config.py` 中将 `sslmode=require` 转为 `connect_args["ssl"] = "require"` |
| 密码安全 | 使用加盐哈希保存密码 | 不直接保存明文密码，而是保存 `salt + PBKDF2-HMAC-SHA256` 结果 | `backend/utils/security.py` 中 `pbkdf2_hmac("sha256", ..., 100_000)` |
| 防时序攻击 | 常量时间比较 | 校验密码时使用 `hmac.compare_digest`，降低比较过程泄露风险 | `verify_password()` |
| 会话鉴权 | Bearer Token 校验 | 只有持有合法令牌的用户才能访问需要登录的接口 | `backend/utils/auth.py` 中对 token 格式和有效性进行校验 |
| 账号唯一性 | 唯一索引防止重复账号 | 防止多个用户注册相同用户名，避免身份混淆 | `users.username` 上有唯一索引 `ix_users_username` |
| 令牌唯一性 | 唯一索引防止重复 token | 降低 token 冲突与错误复用风险 | `user_tokens.token` 和 `user_tokens.user_id` 都设置唯一约束 |
| 错误信息收敛 | 统一异常处理 | 返回通用数据库错误，避免把底层 SQL 细节直接暴露给前端 | `backend/utils/exception.py` 中 `Integrity error`、`Database error` |
| 权限分层 | 角色表 + 业务侧判断 | 区分学生、教师、管理员，控制不同功能访问范围 | `roles` 表和路由中 `role_id` 判断 |

### 2.1 数据库安全层面的优点

- 认证数据与业务数据分离，`user_tokens` 独立成表，便于令牌失效、替换和审计。
- 密码采用随机盐哈希存储，即使数据库泄露，也不会直接暴露用户明文密码。
- 数据库连接支持 SSL，适合云数据库场景下的远程传输安全。
- 唯一索引让账号和 token 具备天然的数据库级约束，而不只依赖应用层判断。

### 2.2 数据库安全层面的不足与例子

- 默认管理员账号采用固定用户名 `admin` 和默认密码 `admin123456` 自动补齐，便于初始化，但如果部署后不立即修改，会带来较大安全风险。
- token 目前以明文形式存储在 `user_tokens` 表中，若数据库被直接读取，攻击者可拿到有效 token 继续访问系统；更稳妥的方案是仅保存 token 摘要或改用 JWT 加签并缩短有效期。
- 当前权限控制主要在应用层完成，数据库本身没有进一步采用 PostgreSQL 原生角色、视图隔离或行级安全策略；也就是说，若绕过应用直连数据库，数据库层面对“学生不能看教师数据”的限制并不充分。
- `auth_crud.create_token()` 将 token 有效期设置为 7 天，使用方便，但有效时间偏长，属于可用性优先于安全性的取舍。

## 3. 数据库完整性技术分析

| 完整性类型 | 项目体现 | 说明 | 具体例子 |
| --- | --- | --- | --- |
| 实体完整性 | 主键约束 | 每张核心实体表都具有主键，保证记录可唯一标识 | `users.user_id`、`courses.course_id`、`assignments.assignment_id` |
| 参照完整性 | 外键约束 | 子表记录必须引用存在的父表记录 | `assignment_submissions.user_id -> users.user_id`，`assignment_submissions.assignment_id -> assignments.assignment_id` |
| 联系完整性 | 联合主键联系表 | 多对多关系通过联合主键联系表避免重复关系 | `teaching_relations(user_id, course_id)`、`resource_tags(resource_id, tag_id)` |
| 唯一完整性 | 唯一约束/索引 | 保证业务上的“不重复”规则 | `course_chapters` 上 `unique(course_id, chapter_order)` |
| 非空完整性 | `nullable=False` | 强制必填字段不能缺失 | 如 `roles.role_name`、`ai_messages.message_content`、`notes.content` |
| 默认值完整性 | `server_default` / `default` | 确保时间戳、状态字段在缺省情况下仍有有效值 | `created_at = now()`、`session_status = "进行中"` |
| 事务完整性 | 提交/回滚机制 | 异常时回滚事务，避免部分写入成功、部分失败 | `get_db()` 中捕获异常后 `rollback()` |
| 删除一致性 | ORM 级联删除 | 删除父对象时清理子对象，降低孤儿记录风险 | `Course` 删除时会级联删除章节、作业、笔记等关联数据 |

### 3.1 完整性设计的具体例子

- 章节顺序完整性：`course_chapters` 设置 `unique(course_id, chapter_order)`，因此同一门课内不能同时出现两个“第 1 章”。这能防止目录结构混乱。
- 多对多关系完整性：`resource_tags` 的 `(resource_id, tag_id)` 既是联合主键又是双外键，因此不会出现“同一资源重复绑定同一标签”的记录，也不会出现绑定到不存在资源或不存在标签的情况。
- 作业提交完整性：`assignment_submissions` 通过双外键保证提交记录一定隶属于某个真实作业和某个真实用户。
- AI 会话消息完整性：`ai_messages.session_id` 外键指向 `ai_sessions`，保证消息必须附着在某个真实会话上，不能凭空存在。

### 3.2 当前数据库完整性方面仍存在的问题

- `assignment_submissions` 没有设置 `unique(assignment_id, user_id)`。如果业务规则是一名学生对同一作业只保留一条最终提交记录，那么当前数据库层面无法阻止重复提交行。例如同一个学生可能在同一作业下生成两条“已提交”记录，这会增加批改和统计复杂度。
- `learning_progress` 使用 `progress_type + target_id` 表示不同类型的学习目标，但 `target_id` 并未对具体目标表建立外键。这是一种灵活但弱参照的设计。例子是：当 `progress_type="chapter"` 时，`target_id` 应该指向章节；当 `progress_type="resource"` 时，`target_id` 应该指向资源。数据库本身无法验证 `target_id=12` 到底是否真的存在于对应目标表中。
- 许多状态字段采用字符串保存，例如 `submission_status`、`session_status`、`enrollment_status`，但数据库层没有设置 `CHECK` 约束或枚举类型，因此理论上可以写入非法状态值，如“未知状态”。
- `assignment_submissions.score`、`users.age` 等字段缺少数据库级范围约束。例如分数字段理论上可以被写成 `-5` 或 `1000`，这说明当前域完整性更多依赖应用层而不是数据库层。

## 4. 模式及范式分析

### 4.1 模式分析

- 本项目采用的是典型的关系数据库模式，以实体表、联系表和扩展表三类结构组织数据。
- 实体表包括 `users`、`roles`、`courses`、`course_chapters`、`course_resources`、`assignments`、`notifications`、`ai_sessions` 等，用于保存核心业务对象。
- 联系表包括 `teaching_relations`、`enrollment_relations`、`resource_tags`，用于处理多对多关系。
- 扩展表包括 `resource_versions`、`resource_chunks`、`assignment_submissions`、`ai_messages`、`ai_memories`，用于承载版本、切片、提交记录和 AI 扩展数据。
- 从实现看，系统已具备较清晰的“用户权限域、课程资源域、学习支持域、作业域、AI 域”分层，说明数据库模式具备较好的模块化特征。

### 4.2 范式分析

| 范式 | 项目符合情况 | 原因说明 | 例子 |
| --- | --- | --- | --- |
| 第一范式（1NF） | 基本满足 | 字段大多保持原子性，没有把“多个值”直接塞进普通标量字段中 | `users.name`、`courses.course_name`、`assignments.assignment_content` 都是一列一值 |
| 第二范式（2NF） | 基本满足 | 含联合主键的联系表中，非主属性都依赖完整主键而非部分主键 | `enrollment_relations` 中 `enrolled_at`、`enrollment_status` 依赖 `(user_id, course_id)` 这一整体关系 |
| 第三范式（3NF） | 大体满足，是当前设计主目标 | 大部分非主属性只依赖主键，不依赖其他非主属性 | `users` 中角色说明被拆到 `roles`，避免把 `role_name`、`permission_description` 反复冗余在用户表中 |
| BCNF | 多数核心表接近满足，但不是全库强制追求 | 对教学业务主数据已较规范，但对 AI 扩展和多态进度结构做了工程化取舍 | `learning_progress` 中 `target_id` 的含义依赖 `progress_type`，这种隐含依赖不如纯 BCNF 结构严格 |
| 第四范式（4NF） | 无需全面追求，关键多值关系已拆分 | 多对多、多值依赖已通过联系表分解，已满足主要需求 | `resource_tags` 已将“资源-标签”的多值依赖拆开 |
| 第五范式（5NF） | 未进一步追求 | 再继续分解会显著增加连接复杂度，而当前业务不存在复杂连接依赖问题 | 教学系统主数据没有必要拆成更细碎的可逆连接结构 |

### 4.3 为什么不用更高范式

- 不继续全面追求 BCNF、4NF、5NF，主要不是理论上做不到，而是从工程收益上看没有必要。
- 对于 `roles`、`users`、`courses`、`assignments`、`resource_tags` 这类典型教学业务表，当前 3NF 设计已经能较好消除冗余、避免插入异常和更新异常。
- 如果继续把 AI 扩展数据完全拆成更多子表，例如把 `ai_messages.message_content` 的 JSON 结构拆成若干消息片段表、附件表、工具调用表，理论上范式会更高，但会导致：
  - 表数量明显增加；
  - 查询链路变长；
  - 插入一次消息需要多次写表；
  - AI 消息结构变化时数据库也要频繁改表。
- 因此，项目对 AI 数据采用了 `JSONB` 和向量字段等半结构化设计，本质上是“规范化与灵活性之间”的折中。

### 4.4 是否因为性能做了取舍

- 是，项目在部分地方明显做了性能和灵活性优先的取舍，但整体仍以 3NF 为主。
- `ai_messages.message_content` 使用 `JSONB` 而不是继续拆表，优点是单次读写更直接、适配 AI 消息结构更灵活，缺点是字段级约束能力变弱，范式纯度下降。
- `ai_memories.memory_meta` 使用 `JSONB`、`embedding` 使用向量字段，是为了支持检索增强和个性化记忆。如果完全按传统关系模式拆开，会让相似度检索和上下文恢复变得更复杂，也不利于性能。
- `learning_progress` 使用 `progress_type + target_id` 的通用模式，而不是为章节进度、资源进度、作业进度分别建三张表，这是为了减少表数量、降低接口复杂度、提升开发效率；但代价是数据库无法为 `target_id` 建立严格外键，因此完整性和范式严格性有所牺牲。
- `resource_chunks` 将分块结果独立成表，而不是把所有切片内容仍塞在资源主表中，这是兼顾性能和规范化的正向取舍。这样可以避免 `course_resources` 过宽、过大，也便于后续分块检索。

## 5. 综合结论

- 从框架和技术角度看，该项目采用了较典型的 Python 全栈方案：FastAPI 负责后端接口，Streamlit 负责前端，PostgreSQL 负责核心数据存储，SQLAlchemy 负责关系建模，LangChain 和 pgvector 提供 AI 扩展能力。
- 从数据库安全角度看，项目已经具备密码加盐哈希、Bearer Token、唯一索引、SSL 连接和统一异常处理等基础能力，适合作为课程设计实现方案。
- 从数据库完整性角度看，项目较好地实现了主键、外键、唯一约束、非空约束和事务回滚，整体结构比较规范，但在“数据库原生约束”方面还可以继续加强。
- 从范式角度看，项目整体处于以 3NF 为主、部分核心表接近 BCNF、AI 扩展模块适度反规范化的状态，这种设计符合“教学管理 + AI 扩展”的实际场景。
- 如果后续继续优化，建议优先补充以下数据库级约束：`assignment_submissions(assignment_id, user_id)` 唯一约束、状态字段 `CHECK` 约束、分数字段范围约束，以及更严格的 token 存储与默认管理员安全策略。
